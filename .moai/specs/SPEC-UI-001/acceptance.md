# SPEC-UI-001 인수 기준: Windows GUI Application (CustomTkinter)

## 1. 인수 기준 개요

이 문서는 SPEC-UI-001의 모든 요구사항에 대한 검증 시나리오를 정의한다. Given-When-Then 형식을 사용하며, 각 시나리오는 하나 이상의 요구사항과 연결된다.

---

## 2. 테스트 시나리오

### 시나리오 1: 애플리케이션 실행 및 다크 테마 (REQ-U-001)

```gherkin
Feature: 애플리케이션 실행 및 기본 테마

  Scenario: GUI 애플리케이션이 다크 테마로 실행된다
    Given pdf-tool-gui 실행 파일이 존재한다
    When 사용자가 pdf-tool-gui를 실행한다
    Then CustomTkinter 기반 메인 윈도우가 표시된다
    And 기본 테마는 다크 모드이다
    And 사이드바에 8개 작업 메뉴(Cut, Merge, Split, Rotate, Resize, Compress, Watermark, Info)가 표시된다

  Scenario: 사이드바 네비게이션으로 페이지를 전환한다
    Given GUI 애플리케이션이 실행된 상태이다
    When 사용자가 사이드바에서 "Merge" 버튼을 클릭한다
    Then 메인 콘텐츠 영역에 Merge 작업 페이지가 표시된다
    And 이전에 표시되던 페이지는 숨겨진다
    And "Merge" 사이드바 버튼이 활성 상태로 하이라이트된다
```

### 시나리오 2: 드래그 앤 드롭 파일 로딩 (REQ-E-001, REQ-N-002)

```gherkin
Feature: 드래그 앤 드롭으로 PDF 파일 로딩

  Scenario: 유효한 PDF 파일을 드래그 앤 드롭한다
    Given Cut 작업 페이지가 표시된 상태이다
    When 사용자가 유효한 PDF 파일을 파일 입력 영역에 드래그 앤 드롭한다
    Then 해당 PDF 파일이 로드된다
    And 파일 이름과 페이지 수가 표시된다
    And 작업 실행 버튼이 활성화된다

  Scenario: 지원하지 않는 파일 형식을 드래그 앤 드롭한다
    Given Cut 작업 페이지가 표시된 상태이다
    When 사용자가 .docx 파일을 파일 입력 영역에 드래그 앤 드롭한다
    Then 에러 메시지가 표시된다: "PDF 파일만 지원합니다"
    And 파일은 로드되지 않는다
    And 작업 실행 버튼은 비활성 상태를 유지한다

  Scenario: 파일 선택 다이얼로그로 PDF를 선택한다
    Given Cut 작업 페이지가 표시된 상태이다
    When 사용자가 "파일 선택" 버튼을 클릭한다
    Then OS 파일 선택 다이얼로그가 표시된다
    And 파일 필터가 "PDF 파일 (*.pdf)"로 설정되어 있다
```

### 시나리오 3: GUI를 통한 PDF Cut 작업 실행 (REQ-E-002, REQ-S-001)

```gherkin
Feature: GUI에서 PDF Cut 작업 실행

  Scenario: Cut 작업을 성공적으로 실행한다
    Given Cut 작업 페이지에서 PDF 파일이 로드된 상태이다
    And 페이지 범위가 "1-3, 5"로 입력된 상태이다
    When 사용자가 "실행" 버튼을 클릭한다
    Then 프로그레스 바가 표시된다
    And 실행 버튼이 비활성화된다
    And 작업이 완료되면 성공 메시지가 표시된다
    And 출력 파일 경로가 결과 영역에 표시된다
    And 실행 버튼이 다시 활성화된다

  Scenario: 파일 미로드 상태에서 실행을 시도한다
    Given Cut 작업 페이지가 표시된 상태이다
    And PDF 파일이 로드되지 않은 상태이다
    When 사용자가 "실행" 버튼을 확인한다
    Then 실행 버튼은 비활성 상태이다
    And "PDF 파일을 먼저 선택하세요" 안내 메시지가 표시되어 있다
```

### 시나리오 4: 에러 핸들링 (REQ-N-003, REQ-C-001)

```gherkin
Feature: 에러 처리 및 안정성

  Scenario: 작업 실행 중 에러가 발생한다
    Given Cut 작업 페이지에서 손상된 PDF 파일이 로드된 상태이다
    When 사용자가 "실행" 버튼을 클릭한다
    Then 에러 다이얼로그가 표시된다
    And 에러 메시지에 문제 원인이 포함되어 있다
    And 프로그레스 바가 초기화된다
    And 실행 버튼이 다시 활성화된다
    And 애플리케이션은 정상 상태로 복귀한다

  Scenario: 작업 진행 중 창 닫기를 시도한다
    Given PDF 작업이 진행 중인 상태이다
    When 사용자가 창의 닫기(X) 버튼을 클릭한다
    Then 확인 다이얼로그가 표시된다: "작업이 진행 중입니다. 종료하시겠습니까?"
    And 사용자가 "예"를 선택하면 작업이 중단되고 애플리케이션이 종료된다
    And 사용자가 "아니오"를 선택하면 다이얼로그가 닫히고 작업이 계속된다

  Scenario: 원본 파일 덮어쓰기 방지
    Given Cut 작업 페이지에서 PDF 파일이 로드된 상태이다
    And 출력 경로가 원본 파일과 동일하게 설정된 상태이다
    When 사용자가 "실행" 버튼을 클릭한다
    Then 경고 다이얼로그가 표시된다: "원본 파일을 덮어쓸 수 없습니다"
    And 작업은 실행되지 않는다
```

### 시나리오 5: Merge 작업 다중 파일 처리 (REQ-S-003)

```gherkin
Feature: Merge 작업 다중 파일 관리

  Scenario: 다중 PDF 파일을 추가하고 순서를 조정한다
    Given Merge 작업 페이지가 표시된 상태이다
    When 사용자가 3개의 PDF 파일을 추가한다
    Then 파일 목록에 3개 파일이 표시된다
    And 각 파일 옆에 위로/아래로 이동 버튼이 표시된다
    And 각 파일 옆에 제거 버튼이 표시된다

  Scenario: 파일 순서를 변경한 후 Merge를 실행한다
    Given Merge 작업 페이지에 3개 파일이 추가된 상태이다
    When 사용자가 두 번째 파일을 첫 번째로 이동시킨다
    And 사용자가 "실행" 버튼을 클릭한다
    Then 변경된 순서대로 PDF 파일이 병합된다
    And 성공 메시지와 출력 파일 경로가 표시된다
```

### 시나리오 6: 테마 전환 (REQ-O-001)

```gherkin
Feature: 다크/라이트 테마 전환

  Scenario: 라이트 테마로 전환한다
    Given 애플리케이션이 다크 테마로 실행된 상태이다
    When 사용자가 테마 전환 토글을 클릭한다
    Then 애플리케이션 전체가 라이트 테마로 전환된다
    And 모든 위젯의 색상이 라이트 테마에 맞게 변경된다
```

## 3. 품질 게이트 기준

### 3.1 기능 완성도

| 항목 | 기준 |
|------|------|
| 8개 작업 페이지 구현 | 모든 페이지가 동작하며 기존 commands 함수를 정상 호출 |
| 드래그 앤 드롭 | Windows 환경에서 PDF 파일 드롭이 정상 동작 |
| 프로그레스 표시 | 모든 작업에서 진행 상태가 표시되고 UI가 멈추지 않음 |
| 에러 핸들링 | 모든 예외 상황에서 에러 다이얼로그 표시 및 안정 복귀 |

### 3.2 비기능 요구사항

| 항목 | 기준 |
|------|------|
| 애플리케이션 시작 시간 | 5초 이내 |
| UI 응답성 | 작업 실행 중에도 UI가 멈추지 않음 |
| EXE 파일 크기 | 50MB 이하 |
| 메모리 사용량 | 유휴 시 200MB 이하 |

### 3.3 코드 품질

| 항목 | 기준 |
|------|------|
| GUI 모듈 테스트 | 위젯 단위 테스트 및 통합 테스트 |
| 기존 테스트 통과 | 기존 17개 테스트 파일 전체 통과 (회귀 없음) |
| 타입 힌트 | 모든 공개 함수에 타입 힌트 적용 |
| 문서화 | 모듈 및 클래스 docstring 작성 |

## 4. 검증 방법

### 4.1 수동 테스트

- Windows 10/11 환경에서 GUI 실행 확인
- 각 작업 페이지에서 PDF 파일 처리 시나리오 실행
- 드래그 앤 드롭 기능 확인
- 에러 시나리오 재현 및 복구 확인

### 4.2 자동화 테스트

- `pytest`를 이용한 위젯 단위 테스트
- 비즈니스 로직 통합 테스트 (commands 함수 호출 검증)
- PyInstaller 빌드 후 EXE 실행 검증

### 4.3 빌드 검증

- PyInstaller로 단일 EXE 빌드 성공
- EXE 파일 크기 50MB 이하 확인
- 빌드된 EXE에서 모든 기능 동작 확인
